# Schema codegen — keeping clients in sync with the database

The database schema is the single source of truth for the shape of a run, route, integration, and user profile. Both clients — the SvelteKit web app and the Flutter mobile apps — derive their row types from it automatically. Change the schema, regenerate, done.

This doc covers:
- [Why](#why)
- [What gets generated](#what-gets-generated)
- [The workflow](#the-workflow)
- [Testing the generators](#testing-the-generators)
- [Testing drift detection end-to-end](#testing-drift-detection-end-to-end)
- [Troubleshooting](#troubleshooting)

Phase 1 of "Cross-platform parity enforcement" in [roadmap.md](roadmap.md) is the initiative this flow comes from. Phases 2 (parity matrix) and 3 (cross-client integration test) are not built yet.

## Why

Before codegen, each client hand-wrote its own row-to-model mapping:

- `ApiClient._runFromRow` in Dart — reached into `row['distance_m']`, `row['started_at']`, etc.
- `Run` interface + `fetchRunById` in TypeScript — re-declared every column by hand.

Any time the schema changed — adding `track_url`, dropping `track`, renaming a column, adding `surface` on routes — one client would pick up the change and the other would silently drift. We lost hours several times chasing bugs caused by exactly this: `activity_type` stored on mobile but not displayed on web, `surface` stored by web but dropped by mobile, `moving_time` computed on one platform and not the others. Each one was a manual hunt.

The fix is to make the schema-to-code mapping a build step, not a human step. Rename a column in a migration → regenerate → compile error at every reader that referenced the old name. Drift becomes impossible to merge without noticing.

## What gets generated

### Web — `apps/web/src/lib/database.types.ts`

Generated by `supabase gen types typescript --local`. ~370 lines. Contains:

- `Database['public']['Tables']['runs']['Row']` — full row shape for every table, auto-derived from the live local Supabase instance.
- `Database['public']['Tables']['runs']['Insert']` and `['Update']` variants with optional defaults.
- `Database['public']['Views']` and `Database['public']['Functions']` for materialized views and RPCs.
- The `Json` type alias.

`apps/web/src/lib/types.ts` then derives the app-facing types from the generated ones:

```ts
import type { Database } from './database.types';

type RunRow = Database['public']['Tables']['runs']['Row'];

export type Run = Omit<RunRow, 'source' | 'metadata'> & {
	source: RunSource;                          // narrow from string
	metadata: Record<string, unknown> | null;   // loosen from Json
	track: TrackPoint[] | null;                 // lazy-loaded, not a column
};
```

The `Omit<...> & { ... }` pattern is deliberate: start from what the DB gives us, override only the fields where the client has better knowledge than the schema.

### Mobile — `packages/core_models/lib/src/generated/db_rows.dart`

Generated by `scripts/gen_dart_models.dart`. Contains one class per table in the `_tables` allowlist (`runs`, `routes`, `integrations`, `user_profiles`, `route_reviews`):

```dart
class RunRow {
  static const String table = 'runs';
  static const String colId = 'id';
  static const String colUserId = 'user_id';
  static const String colStartedAt = 'started_at';
  // ...one const per column...

  final String id;
  final String userId;
  final DateTime startedAt;
  // ...one final field per column, snake_case preserved as camelCase...

  const RunRow({ required this.id, required this.userId, /*...*/ });
  factory RunRow.fromJson(Map<String, dynamic> json) => /*...*/;
  Map<String, dynamic> toJson() => /*...*/;
}
```

`ApiClient` uses these rows as the wire format. It constructs a `RunRow` from a domain `Run` before upserting, and runs `RunRow.fromJson` before building a domain `Run` back. Column names are always referenced through the generated constants — `RunRow.colStartedAt`, never the string literal `'started_at'` — so a rename propagates as a compile error.

The rich `Run` / `Route` / `Waypoint` domain classes in `core_models/lib/src/` are kept — they hold `Duration`, `DateTime`, the `RunSource` enum, and drop unused columns (`user_id`, `updated_at`). The generated rows are the DTO layer underneath them.

### Wear OS (Kotlin) — `apps/watch_wear/android/app/src/main/kotlin/com/runapp/watchwear/generated/DbRows.kt`

Same generator (`scripts/gen_dart_models.dart`), same parse tree, second emitter. Currently emits only the tables in `_kotlinTables` (just `runs` today — the watch only writes one table). Produces a Kotlin `data class RunRow` with the same column constants (`COL_STARTED_AT`, etc.) and `fromJson` / `toJsonMap` equivalents. `SupabaseClient.saveRun` in `watch_wear` references the generated constants, same pattern as the Dart `ApiClient`. A migration that drops or renames a column regenerates both the Dart file and the Kotlin file, and either consumer fails to compile if it references the old name.

If you add a new table that the Wear OS app needs to write, add it to `_kotlinTables` in `scripts/gen_dart_models.dart` and rerun. The Kotlin type mapping is narrower than Dart's `dynamic` — jsonb columns become `kotlinx.serialization.json.JsonElement`, so callers use the JSON tree API to read nested values.

Kotlin has no `parity-types`-equivalent CI gate yet; rely on `./gradlew compileDebugKotlin` locally catching drift. Adding a CI check that regenerates the Kotlin file and fails on uncommitted diff is a TODO — same shape as the TS `gen:types:check` job.

## The workflow

### When you change the schema

```bash
cd apps/backend

# 1. Author the migration
supabase migration new add_activity_type_to_runs
$EDITOR supabase/migrations/<timestamp>_add_activity_type_to_runs.sql

# 2. Apply it locally
supabase db reset

# 3. Regenerate both row files
cd ..
npm run gen:types                       # writes apps/web/src/lib/database.types.ts
dart run scripts/gen_dart_models.dart   # writes packages/core_models/lib/src/generated/db_rows.dart AND
                                        # apps/watch_wear/android/app/src/main/kotlin/com/runapp/watchwear/generated/DbRows.kt

# 4. Build the clients to find every drift site
cd apps/web && npm run check
cd ../.. && melos exec -- dart analyze
```

Steps 3 and 4 are where the drift protection pays off. If the schema change breaks a consumer, the compiler names the file and line.

### When you pull someone else's schema change

Exactly the same — run both generators, then build. The regenerated files are committed to the repo, but your local Supabase instance needs to be at the same migration state as theirs:

```bash
cd apps/backend
supabase db reset    # replays all migrations, including theirs
cd ../..
npm run gen:types:check   # will pass if their committed types match yours
```

## Testing the generators

### Unit-test the Dart generator

There isn't one yet — the generator lives in `scripts/` and has no test suite. For now, "does it work?" means:

```bash
dart run scripts/gen_dart_models.dart
```

This re-reads all migrations, re-emits the file, and overwrites `db_rows.dart`. If the generator throws, the migration SQL hit an unsupported form (a CHECK constraint that confuses the column parser, a `create extension`, etc.) — grow `_parseCreateTable` / `_parseAlterTable` in `scripts/gen_dart_models.dart` as needed. The parser intentionally covers only `create table`, `alter table ... add/drop column`. Everything else (`create index`, `create policy`, `insert into storage.buckets`, `$$...$$` function bodies) is silently ignored.

A regression suite will become worth building if the SQL subset grows beyond what can be eyeballed in one read-through. Until then, the fact that the generator output is committed and CI builds with it is the test.

### Smoke-test it against the committed migrations

```bash
# Regenerate and diff against what's in the repo
dart run scripts/gen_dart_models.dart
git diff packages/core_models/lib/src/generated/db_rows.dart
```

A clean run produces no diff. If it produces a diff, either:
1. You forgot to commit the last regeneration (commit the delta now).
2. Someone edited `db_rows.dart` by hand (don't — it says "GENERATED FILE — DO NOT EDIT" at the top).

### Smoke-test the TypeScript generator

```bash
# Requires `supabase start` in apps/backend first
cd apps/backend
npm run gen:types:check
echo "exit=$?"
```

Exit 0 = in sync. Non-zero = drift; `gen:types:check` prints a unified diff of what changed.

## Testing drift detection end-to-end

The real test — prove that the generators actually catch drift — is to introduce a breaking change and watch the build blow up. Here's a non-destructive way to do it:

### 1. Rename a column in a scratch migration

```bash
cd apps/backend
supabase migration new scratch_rename_distance
```

Put this in the new migration file:

```sql
alter table runs rename column distance_m to total_distance_m;
```

Apply it:

```bash
supabase db reset
```

### 2. Regenerate both row files

```bash
cd ../..
npm run gen:types
dart run scripts/gen_dart_models.dart
```

Both files should regenerate cleanly (the generators only care that the SQL parses, not whether downstream code still compiles).

### 3. Watch the clients fail to build

```bash
# Web — expect a type error in data.ts, mock-data.ts, or wherever distance_m is referenced
cd apps/web && npm run check
```

Expect errors like:

```
src/lib/mock-data.ts:20:55 - Property 'distance_m' does not exist on type 'Run'.
```

```bash
# Dart — expect analysis errors in api_client where RunRow.colDistanceM used to live
cd ../.. && melos exec -- dart analyze --fatal-warnings 2>&1 | grep error
```

Expect errors like:

```
error - The getter 'distanceM' isn't defined for the class 'RunRow'.
error - Undefined name 'colDistanceM'.
```

These are the bugs the system is supposed to catch. Seeing them proves the safety net is intact.

### 4. Roll back the scratch migration

**This is important** — don't commit the scratch migration. Delete it and reset:

```bash
rm apps/backend/supabase/migrations/*_scratch_rename_distance.sql
cd apps/backend && supabase db reset
cd ../..
npm run gen:types
dart run scripts/gen_dart_models.dart
git status   # should show no changes to the committed type files
```

If `git status` does show a diff in `database.types.ts` or `db_rows.dart`, your rollback is incomplete.

### 5. Try the CI drift check

Still at clean state, verify the CI job passes locally:

```bash
cd apps/backend
npm run gen:types:check  # exit 0
```

Now deliberately hand-edit `apps/web/src/lib/database.types.ts` (change one type annotation) and re-run:

```bash
npm run gen:types:check  # exit non-zero, prints a diff
```

This is exactly what CI runs in the `parity-types` job. Revert the hand-edit when you're done.

## Troubleshooting

### `gen:types` puts "Connecting to db 5432" at the top of the file

Fixed — the `gen:types` script pipes through `grep -v '^Connecting to db'` to strip that stderr-on-stdout line. If it reappears, the CLI changed its output format; check `supabase gen types typescript --help` and adjust the filter.

### `gen:types` produces an empty file

`supabase start` isn't running in `apps/backend`, or the local DB is broken. Check `supabase status` and `supabase db reset`.

### `gen_dart_models.dart` throws `Unmatched ( in ...`

The migration SQL has an unbalanced paren — either a real syntax error in the migration or a form the parser doesn't understand (nested subqueries in a column default, rare). Fix the migration, or special-case the statement in `scripts/gen_dart_models.dart`.

### `dart analyze` fails on `db_rows.dart` after regeneration

The generator emitted something the analyser rejects. Most likely causes:
- A new Postgres type the generator doesn't have a Dart mapping for — add it to `_pgToDart` in `scripts/gen_dart_models.dart`.
- A column name that isn't a valid Dart identifier once camelCased — rename the column in SQL.

### The TS compiler complains that a row field is missing after regeneration

Supabase only marks a column as optional in the `Row` type when it has a database default. If you add a nullable column without a default, it'll be `string | null` but *required* in the Row — you must include it when constructing mock data. This happened to `mock-data.ts` during the initial rollout; the fix was adding `updated_at: <timestamp>` to the mock row helpers.

### The "narrow union" columns (`source`, `surface`, `provider`) drifted out of sync with reality

The generated types see them as `string` because Postgres has no CHECK constraint on those columns. `apps/web/src/lib/types.ts` re-narrows them via `Omit<...> & { source: RunSource; ... }`. If a client starts writing a new value (e.g. `'zwift'`) the narrow union won't include it. Two fixes, pick one:
1. Add the new value to the union in `types.ts` (fast, client-side only).
2. Add a CHECK constraint to the column in a migration, regenerate, delete the override in `types.ts` (slower, enforced at the DB).

Same pattern applies to `IntegrationProvider`, `RouteSurface`, `PreferredUnit`, `SubscriptionTier`.

## What this does not solve

- **Ephemeral / undeclared fields inside `jsonb` columns** — `runs.metadata` is `jsonb` in the DB and `Record<string, unknown>` / `Map<String, dynamic>` on both clients. If mobile writes `metadata.activity_type` and web doesn't read it, no generator will catch that. The roadmap's Phase 3 cross-client integration test is designed to cover this.
- **Semantic parity** — generators catch "this column exists", not "this column means the same thing on both platforms". `moving_time` computed differently on web vs Android is still a human-review problem.
- **UI parity** — whether both platforms actually *show* a field is covered by the Phase 2 parity matrix, not codegen.

Drift protection is one layer. Both Phase 2 and Phase 3 of the roadmap stack on top of it.
